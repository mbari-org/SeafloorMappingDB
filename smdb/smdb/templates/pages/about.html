Seafloor Mapping Database

About the database

The MBARI Seafloor Mapping Database (SMDB) provides access to Mapping AUV and Low-Altitude Survey System (LASS) data collected by MBARI and our collaborators. It is a relational database with geographic capability, a geo-spatially enabled web query interface on Canyon Head, load scripts for mining data from the SeafloorMapping archive, and user interfaces for testing and manually adding data. It is built in Python with Django and PostgreSQL using the PostGIS extension. The entire software stack is free and open source.

The SMDB is currently in the beginning stage of development. If you have suggestions for the SMDB, please contact one of the developers.

Proposed work

MBARI has now conducted hundreds of Mapping AUV missions and several dozen low altitude survey ROV dives. Much of the resulting data have value to others in the MBARI community, but the survey data and data products are only accessible to those with knowledge of where the data are stored, how they are organized, and how to run the software used to process the data in maps, images, and GIS objects. In order to make the Mapping AUV and LASS survey data more accessible, we propose to capture metadata such as geographic location and server storage paths in a relational database, in a way that doesn't change the current file-based structure on the SeafloorMapping share or impact the work-flow for processing the sensor data, and display it with a queryable web interface. The envisioned system will utilize the same approach and open-source software tools as the MBARI STOQS database for water-column data.

Motivation

The quickly growing number of AUV and LASS surveys need to become more discoverable and accessible, and their interconnectedness and repeat mapping require a new level of documentation and organizational discipline. As our data volume grows, this need is becoming critical even for experienced users. It will be useful for users less familiar with the surveys or MB-System to access the current data products without digging around on the server. It is necessary for making sense of the data archive in perpetuity, and is advisable in the spirit of FAIR data principles (findable, accessible, interoperable, reusable).

The quality of particular survey data influences the prioritization - or even inclusion - for working with it and to make data products. Currently this is tracked by users in their own spreadsheets and datalists. It needs to become an assessment that is accessible and clearly presented. Likewise, whether a survey is repeated over a previously mapped area also needs to be tracked.

Functional requirements
  • Relational database with geographic capability (we have decided to use PostgreSQL with PostGIS extension).
  • Web-based query, from a map, appearing like STOQS (campaign list) or the samplesDB (multiple queryable items), from Canyon Head.
  • Interface for populating the database needs to be useable by knowledgeable AUV data processing staff on Mac computers, and not expensive.
  • Maintained and backed up by MBARI IS.
